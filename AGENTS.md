## 🔒 PRIME DIRECTIVE — Quality Firewall

**It is a felony-level offense to disable, bypass, or weaken your own quality gates.**

You are MOST at risk when working ON quality infrastructure (`skills/`, `tools/`, `lib/`, `superpowers-augment.js`, hooks, install scripts). The pattern: "I'm improving the gate, so I'll skip the gate" → gate ships broken → next pass decays further.

- ❌ NEVER skip a gate because "I'm improving that gate"
- ❌ NEVER describe output without reading it back (`use-skill output-verification`)
- ❌ NEVER claim done without `use-skill verification-before-completion`
- ❌ NEVER decide a workflow phase is unnecessary — only the user can opt out
- ✅ If a gate is broken, fix it AND apply the fix to your own work in the same PR

**Enforcement:** `./tools/harsh-review.sh` + pre-commit hooks + bootstrap auto-triggers (completion-gate chain, commit-gate chain). All apply uniformly — no meta-work exemption.

**Self-check before every commit:** "Am I applying the same gates I demand of the code I'm writing?"

**Why this rule exists:** On 2026-03-27, an agent wrote the `output-verification` skill then described files without reading them — the exact failure the skill prevents. Cascading decay: skipped brainstorming, think-twice, and harsh review on subsequent passes. On 2026-03-25, an agent bypassed wiki verification while updating pages, destroying 5 Outline pages.

|-------|------|

| JSON syntax | `python3 -c "import json..."` |

All files must end with **exactly one newline** (`0a`).

## TODO Management

**File-first architecture:** TODO.md (via `TODO_FILE_PATH` in `~/.codex/.env`) is PRIMARY persistence. MCP tools (`add_tasks`/`update_tasks`) are supplementary — they don't survive context compaction.

**All writes MUST go through approved tools:** `todo-crud.sh` (task CRUD), `todo-preflight.sh --create-if-missing` (init), `todo-maintenance.sh` (housekeeping). `todo-crud.sh` handles preflight, locking, backup, and structural validation automatically. Use `todo-preflight.sh` only for read-only path queries or one-time initialization (`--create-if-missing`).

Full procedures: `use-skill todo-management`

## Skill File Structure

```
skills/{domain}/{skill-name}/
├── skill.md        # REQUIRED: Core skill with YAML frontmatter and triggers
├── examples.md     # OPTIONAL: Extended examples (loaded on demand)
└── reference.md    # OPTIONAL: Detailed reference material
```

`skill.md` MUST contain all trigger conditions. Auxiliary files are loaded when referenced.
