## üî¥ Git Workflow ‚Äî Dev / Staging / Production (NON-NEGOTIABLE)

### Branching Model

| Branch | Purpose | Accepts PRs from | Merges into | Cadence |
|--------|---------|-------------------|-------------|---------|
| `dev` | Active development | Feature/fix branches | `staging` | Frequent ‚Äî per feature/fix |
| `staging` | Batch validation | `dev` only | `main` | Deliberate ‚Äî accumulate many changes |
| `main` | Production releases | `staging` only | ‚Äî | **Rare ‚Äî explicit human decision** |

### Feature Initiation

All new feature or fix branches MUST branch from `dev`:

```bash
git fetch origin
git checkout -b feat/my-feature origin/dev
# ... make changes, commit, battery review ...
git push origin feat/my-feature
# Create PR on GitHub targeting dev ‚Üí merge
```

### Promotion Path

```text
feature-branch ‚Üí PR ‚Üí dev
                         ‚Üì (many changes accumulate)
                       staging  ‚Üê explicit human instruction only
                         ‚Üì (rigorous batch review)
                        main    ‚Üê explicit human instruction + review verdict
```

1. **Dev:** Feature/fix branches merge into `dev` via PR. CI must pass. This is the normal cadence ‚Äî frequent, per-change.
2. **Staging:** `dev` merges into `staging` only when **explicitly instructed by the human**. Staging is a deliberate batch-collection point ‚Äî do NOT promote after every feature. Wait until multiple verified changes are ready for release.
3. **Production:** `staging` merges into `main` only when **explicitly instructed by the human** AND a batch code review of all staging changes has been completed and approved.

### Prohibitions

- ‚ùå **NEVER** commit directly to `dev`, `staging`, or `main` ‚Äî always use a branch + PR
- ‚ùå **NEVER** branch features from `main` or `staging` (branch from `dev`)
- ‚ùå **NEVER** promote `dev ‚Üí staging` without explicit human instruction in the current conversation
- ‚ùå **NEVER** promote `staging ‚Üí main` without explicit human instruction + batch review approval
- ‚ùå **NEVER** infer that a single feature landing in `dev` justifies promotion ‚Äî staging must accumulate multiple verified changes. Explicit human instruction to promote a single fix is still allowed.
- ‚ùå **NEVER** treat prior promotion authorization as valid after context compaction or sub-agent handoff ‚Äî the human must restate the instruction in the current active turn
- ‚úÖ **Exception:** Emergency hotfixes may branch from `main`, PR into `main`, then cherry-pick back to `dev`
